# Шаг 1 — Витрины в ClickHouse (предагрегации)

## Цель
Перевести расчеты метрик с «на лету» на предагрегированные витрины, чтобы ускорить ответы API и снять нагрузку с Python.

## Результат шага
- Созданы и заполнены витрины:
  - `point_add.agg_h3_month`
  - `point_add.agg_settlement_month`
  - (опционально) `point_add.points_month`
- Есть процедура пересчета месяца `M` и обновления `data_version`.

## Работы
### 1) Нормализация координат
- Определить правила валидности координат.
- Выделить слой/представление `v_account_geo`:
  - `account_id, lat, lon, settlement_id`
  - дедупликация координат на аккаунт

### 2) Витрина `agg_settlement_month`
- Источник: `v_account_geo` + `point_customer_active/sysblock` + транзакции (payments/charges) в пределах месяца.
- Агрегация: `group by recv_mon, settlement_id`.

### 3) Витрина `agg_h3_month`
- Источник: `v_account_geo` → вычисление `h3_index` на резолюциях (минимум 2 резолюции: для zoom 1–10 и 11–14).
- Агрегация: `group by recv_mon, h3_res, h3_index`.

### 4) (Опционально) `points_month`
- Для ускорения слоя абонентов и упрощения bbox‑фильтрации.

### 5) Оркестрация пересчета
- Скрипт/таска `build_month(M)`:
  - пересчитать витрины месяца `M`
  - записать `data_version`
- Расписание:
  - ежемесячно (закрытие месяца)
  - при необходимости — ежедневный инкремент (если данные догружаются)

## Критерии приемки
- Запросы API на H3/поселки читают только витрины.
- Время ответа на bbox для гексов/поселков p95 < 1s (ориентир).

## Риски/долги
- Подбор `h3_res` потребует калибровки.
- Если поселки будут полигонами — потребуется отдельная гео‑таблица.
