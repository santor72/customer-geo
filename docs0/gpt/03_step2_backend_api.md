# Шаг 2 — Backend API (прод‑контракты, кэш, безопасность)

## Цель
Стабилизировать API: строгая валидация параметров, кэш, версии данных, подготовка под прод.

## Результат шага
- Реализованы контракты `v1`.
- Подключен Redis‑кэш.
- Добавлена авторизация и rate‑limit.

## Работы
### 1) Контракты и валидация
- Валидация:
  - `month` (YYYY-MM)
  - `bbox` (4 числа, диапазоны)
  - `metric` (enum)
  - `zoom`/`h3_res` (enum/диапазоны)

### 2) Версионирование данных
- `GET /api/v1/version` возвращает `data_version`.
- Все ответы слоев включают `data_version`.

### 3) Кэширование
- Redis.
- Ключ включает `data_version`.
- TTL на H3/поселках 1–24h, на subscribers — короткий (например 5–30 мин), или без кэша.

### 4) Безопасность
- JWT (или Keycloak/OIDC) для API.
- Rate limiting на слой subscribers и тяжёлые bbox.
- Отдельный CH user readonly.

### 5) Наблюдаемость
- Метрики:
  - latency p50/p95/p99 по endpoint
  - cache hit rate
  - размер ответов
- Логи запросов (с нормализацией bbox, без персональных данных).

## Критерии приемки
- API стабильно отдает слои из витрин.
- Кэш работает (виден hit rate).
- Неверные параметры дают предсказуемые 400.

## Риски/долги
- Для больших объемов GeoJSON может стать тяжелым — на следующем шаге перейти на MVT.
