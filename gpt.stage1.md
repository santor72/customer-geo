## Главные улучшения (самые “дорогие” по эффекту)

### 1) Уйти от “pandas на лету” к предагрегациям в ClickHouse

Pandas/geopandas/h3 хороши для прототипа, но в проде:

* 8–50k+ точек абонентов и агрегации по месяцам начнут упираться в CPU/память питона.
* Правильнее: **агрегаты H3 и по поселкам считать в ClickHouse** (материализованные представления / агрегирующие таблицы), а Python оставить для API/валидации/форматирования.

**Идея:** все метрики хранятся как “снимки на конец месяца” (`recv_mon`) на нужных пространственных индексах.

### 2) Ввести “единый календарь периода” и строгую модель месяца

Сейчас месяц фигурирует как `recv_mon = последний день месяца`. 
Супер, но нужно закрепить правило:

* в API передаём `month=YYYY-MM` → внутри приводим к `recv_mon = last_day(month)`
* все витрины строятся по `recv_mon`, чтобы не было расхождений.

### 3) Кэш и “тайлы”, иначе фронт будет тяжёлым

Для зумов 1–10 и 11–14 лучше отдавать **агрегаты**, а не сырые точки. Для 15+ — точки, но **в bbox** (по видимой области).

Дальше два варианта:

* Быстро: JSON в bbox + server-side clustering/H3.
* Правильно на будущее: **векторные тайлы (MVT)** для гексов/посёлков/точек.

### 4) Нормализовать справочники (посёлки, координаты) и качество данных

Нужно явно описать:

* что делать с абонентами без координат / с “нулевыми” координатами
* как маппить `ad` (acct_addr_coordinates) → `bx24_location.id` (посёлки) 
* дедупликация `account_id` если бывают несколько координат

## Конкретная архитектура приложения (L1)

### Компоненты

1. **ClickHouse (DWH)**

* Источники как есть.
* * витрины (агрегаты): `agg_h3_month`, `agg_settlement_month`, (опционально) `points_month`.

2. **Backend API (Python)**

* Я бы делал на **FastAPI** (быстрее/удобнее для контрактов и swagger), но можно и Flask как в соглашении. 
* Роли:

  * принимает `month`, `zoom`, `bbox`, `h3_res`
  * отдаёт GeoJSON/JSON (или MVT)
  * авторизация, лимиты, кэширование (Redis)

3. **Redis (cache)**

* Кэш ответов по ключу: `month:bbox:zoom:layer:filters:version`
* TTL (например 1–24 часа) + ручная инвалидация при пересчёте месяца.

4. **ETL/Orchestrator**

* cron / Airflow / Dagster / просто systemd+cron.
* Задача: “пересчитать витрины за месяц M” + обновить версию данных (для кэша).

5. **Frontend**

* SPA: React/Vue + kepler.gl/deck.gl.
* Логика:

  * контрол месяца
  * переключение слоёв по zoom
  * запросы к API по bbox
  * клиентский кэш + debounce

## Схема витрин в ClickHouse (рекомендуемая)

### 1) H3 агрегаты по месяцу

Таблица: `point_add.agg_h3_month`

* `recv_mon Date`
* `h3_res UInt8`
* `h3_index UInt64` (или String, зависит от вашей библиотеки/удобства)
* `active_cnt UInt32`
* `sysblock_cnt UInt32`
* `charges_cnt UInt32`
* `charges_sum Int64/Float64`
* `payments_cnt UInt32`
* `payments_sum Int64/Float64`

Источник:

* координаты абонента → h3_index на нужной резолюции
* `point_customer_active/sysblock` уже по месяцам 
* `dtall` и `pt` агрегируются по месяцу транзакции, но **выводятся в разрезе месяца карты** — значит правило должно быть одно:

  * либо “транзакции в этом календарном месяце”
  * либо “состояние на конец месяца”
  * сейчас у вас “данные выводятся в разрезе месяца на который строится карта”  — я бы формализовал: **для карты месяца M берём транзакции внутри M**.

### 2) Агрегаты по поселкам (точка/полигон)

`point_add.agg_settlement_month`

* `recv_mon Date`
* `settlement_id UInt64` (bx24_location.id)
* `lat/lng Float64` (центр)
* те же метрики

Дальше фронт может рисовать:

* точку поселка (как вы описали)
* а если появятся полигоны — добавите геометрию.

### 3) Точки абонентов (только для zoom 15+)

`point_add.points_month` (опционально, чтобы не джойнить тяжело на лету)

* `recv_mon Date`
* `account_id`
* `lat/lng`
* `settlement_id`
* `is_active`, `is_sysblock`
* `charges_sum_m`, `payments_sum_m` (если нужно прямо на точках)

Если “точки” не нужны как витрина — можно получать их джойном из исходников по bbox, но витрина обычно ускоряет.

## Контракты API (минимально достаточные)

### Справочник периодов

* `GET /api/v1/months` → список доступных `YYYY-MM` (или `recv_mon`)

### Гексагоны (zoom 1–10)

* `GET /api/v1/layers/h3`

  * params: `month=YYYY-MM`, `bbox=minLng,minLat,maxLng,maxLat`, `h3_res=...`
  * response: GeoJSON FeatureCollection (Polygon) или компактный JSON `{h3_index: metrics}`

**Важно:** резолюцию H3 можно выбирать автоматически:

* zoom 1–10 → res 6–7
* zoom 11–14 → res 8–9
  (точные значения подберёте под вашу территорию)

### Посёлки (zoom 1–14)

* `GET /api/v1/layers/settlements`

  * params: `month`, `bbox`
  * response: GeoJSON Point + properties(metrics)

### Абоненты (zoom 15+)

* `GET /api/v1/layers/subscribers`

  * params: `month`, `bbox`, `limit` (и пагинация/курсор)
  * response: точки (GeoJSON или массив)

### Метаданные/версии (для кэша)

* `GET /api/v1/version?month=YYYY-MM` → `{"data_version": "...", "built_at": "..."}`
  Фронт добавляет `data_version` в ключ кэша.

## Логика фронта по zoom (как сделать “приятно”)

Вы уже описали три диапазона.  Я бы уточнил так:

* **Zoom 1–10:**

  * слой H3 “heat” (метрика выбирается: payments_sum / charges_sum / net)
  * слой поселков как точки (чтобы можно было быстро кликать)

* **Zoom 11–14:**

  * H3 на более высокой res *или* выключить и оставить поселки+границы
  * поселки: точки + (опционально) полигоны поселков, если появятся

* **Zoom 15+:**

  * поселки остаются (легкий слой)
  * абоненты: точки по bbox, с ограничением количества и/или динамической подгрузкой

## Пара практичных “добавок”, которые сильно помогут пользователю
